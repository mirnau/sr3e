# Cross-actor economy writes relay through the GM's client; trades require consent from whoever isn't dragging

Purchases and debt payments mutate an actor other than the one running the code (deleting/creating items on the seller, delivering repayment to the creditor) — but Foundry only lets an actor's owner write to it, and the client executing `completePurchase`/`payDebt` is whichever side clicked Confirm, not necessarily the owner of the other actor. Rather than requiring broad ownership grants across players, `sellerMutationRelay.ts` and `debtRepaymentRelay.ts` mirror the existing `messageRelay.ts` ChatMessage pattern: any client that isn't GM emits a socket request instead of mutating directly, and only the GM's client (which has permission everywhere) performs the actual write, each registering its own listener on the shared `system.sr3e` channel rather than folding into combat's `socketHandlers.ts` switch. This requires a GM to be connected for actor-to-actor trades and debt paydowns to complete — consistent with the same requirement the combat contest flow already has.

Separately, whichever side of a trade did *not* perform the drag must explicitly consent via a whispered chat card before anything embeds/deletes — dragging is only ever implicit consent for the dragging side, and the offer/consent cards reuse the same `resolveControllingUser`/`canCurrentUserActFor` functions from `contestCoordinator.ts` that the combat challenge-response flow established. Since players usually can't open each other's character sheets at all, dropping a commodity item onto a token on the shared canvas (`purchaseTokenDropHook.ts`, mirroring `registerMedicalTokenDropHook`) is the primary path for actor-to-actor trades, not sheet-to-sheet dragging.
