# waitForResponse/waitForBothDone cache an early signal instead of only resolving a currently-registered listener

`contestCoordinator.ts`'s `deliverResponse`/`resolveBothDone` used a "signal only if someone's already listening" pattern — if the defender's roll or the "both sides clicked Done" signal arrived before the corresponding `waitForResponse`/`waitForBothDone` call had registered its listener, the signal was silently dropped and the promise never resolved. This was a real, reachable race (`waitForBothDone` is only reached after an awaited `ChatMessage.create()`, a genuine window for the signal to land first), confirmed live: it left `executeProcedure`'s per-actor lock held forever, silently blocking every future roll for that actor until a full page refresh wiped the in-memory lock state. The fix caches the value when no listener is registered yet (`resolvedResponses`/`resolvedBothDone` maps), and `waitForResponse`/`waitForBothDone` check that cache first and resolve immediately if found. Do not revert to the "drop if nobody's listening" version — it looks equivalent in the common case but reintroduces this exact stuck-lock bug under timing that a quick read won't reveal.
