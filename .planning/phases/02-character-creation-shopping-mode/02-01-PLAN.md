# Phase 2 Plan 1: Creation Point Services & Flag Management

**Goal**: Implement services for tracking and calculating creation points (attribute points, skill points) and flag management for character creation state.

## Scope

- `CreationPointsService` - Calculate remaining attribute/skill points based on priority selections
- `AttributeSpendingService` - Handle attribute point spending during character creation
- `SkillSpendingService` - Handle skill point spending (active/knowledge/language categories)
- Flag constant definitions (`isCharacterCreation`, `attributeAssignmentLocked`)
- Point calculation formulas from SR3e rules (30/27/24/21/18 for attributes, 50/40/34/30/27 for skills)

## Dependencies

- Phase 1 complete (CharacterCreationService, CharacterInitializer established)

### Task 2: Implement CreationPointsService

**Action**: Create service to calculate remaining creation points based on priority selections and current character state.

**Files**:
- Create `module/services/character-creation/CreationPointsService.ts`

**Key Methods**:
```typescript
class CreationPointsService {
   static Instance(): CreationPointsService;

   getRemainingAttributePoints(actor: Actor): number;
   getRemainingSkillPoints(actor: Actor, category: "active" | "knowledge" | "language"): number;
   getTotalAttributePoints(priorityLevel: string): number;
   getTotalSkillPoints(priorityLevel: string): number;
   hasUnspentPoints(actor: Actor): boolean;
}
```

**Logic**:
- Attribute points: Read attribute priority from actor flags â†’ map to point total (30/27/24/21/18) â†’ subtract spent points
- Skill points: Read skill priority from actor flags â†’ map to point total (50/40/34/30/27) â†’ distribute across categories â†’ subtract spent points
- Spent points calculated by summing attribute/skill ratings on actor

**Verification**:
- Service follows singleton pattern
- Point calculations match SR3e rules
- Returns correct remaining points for test character

### Task 3: Implement AttributeSpendingService

**Action**: Create service to handle attribute point spending validation and execution during character creation.

**Files**:
- Create `module/services/character-creation/AttributeSpendingService.ts`

**Key Methods**:
```typescript
class AttributeSpendingService {
   static Instance(): AttributeSpendingService;

   canIncreaseAttribute(actor: Actor, attributeKey: string): boolean;
   canDecreaseAttribute(actor: Actor, attributeKey: string): boolean;
   increaseAttribute(actor: Actor, attributeKey: string): Promise<void>;
   decreaseAttribute(actor: Actor, attributeKey: string): Promise<void>;
}
```

**Logic**:
- `canIncrease`: Check remaining points > 0, attribute not at racial max
- `canDecrease`: Check attribute not at racial minimum
- `increase/decrease`: Update actor attribute value, let point calculation happen reactively

**Verification**:
- Cannot increase attribute when no points remain
- Cannot increase beyond racial maximum
- Cannot decrease below racial minimum
- Attribute updates persist to actor

### Task 4: Implement SkillSpendingService

**Action**: Create service to handle skill point spending validation and execution during character creation (active/knowledge/language categories).

**Files**:
- Create `module/services/character-creation/SkillSpendingService.ts`

**Key Methods**:
```typescript
class SkillSpendingService {
   static Instance(): SkillSpendingService;

   canIncreaseSkill(actor: Actor, skillId: string, category: "active" | "knowledge" | "language"): boolean;
   canDecreaseSkill(actor: Actor, skillId: string): boolean;
   increaseSkill(actor: Actor, skillId: string, category: "active" | "knowledge" | "language"): Promise<void>;
   decreaseSkill(actor: Actor, skillId: string): Promise<void>;
}
```

**Logic**:
- `canIncrease`: Check remaining points for category > 0, skill not at max rating
- `canDecrease`: Check skill rating > 0
- `increase/decrease`: Create/update skill item on actor, let point calculation happen reactively
- Category determines which point pool to check (active/knowledge/language have separate pools)

**Verification**:
- Cannot increase skill when no points remain in category
- Cannot decrease skill below 0
- Skill items created/updated correctly on actor
- Different categories use different point pools

## Notes

**Design Patterns**:
- Singleton services with `Instance()` getters (established in Phase 1)
- Services are stateless - all state stored on actor
- Point calculations happen on-demand (no caching)
- Let-it-fail philosophy: no defensive error handling

**SR3e Point Pools**:
- Attribute points: 30 (A), 27 (B), 24 (C), 21 (D), 18 (E)
- Skill points: 50 (A), 40 (B), 34 (C), 30 (D), 27 (E)
- Skill points divide into: active, knowledge, language (proportions TBD)

**Integration Points**:
- CharacterInitializer will need to set `isCharacterCreation` flag on new characters
- StoreManager will subscribe to flags for reactive updates in UI
- Attributes.svelte will call AttributeSpendingService methods
- DicePools.svelte (or Skills component) will call SkillSpendingService methods
