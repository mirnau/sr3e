**Prior decisions affecting this phase:**
- ALL Foundry data reads/writes MUST go through IStoreManager — never direct `document.update()` outside the API layer
- `Creation.ts` schema defines pools as REMAINING values: `activePoints`, `knowledgePoints`, `languagePoints` — do NOT add `*Spent` fields
- Delete-first-then-refund pattern prevents race condition with Foundry's updateActor hook
- Svelte 5 runes syntax ONLY — use `$props()`, `$derived()`, `$effect()`, `$store` subscriptions. No legacy reactive patterns.

**Architecture rule — IStoreManager is the only legal write path:**
`storeManager.GetRWStore(document, path)` returns a Svelte `Writable<T>`.
Writing `$store = value` or `store.set(value)` fires `document.update()` automatically via the store's subscription.
This is the ONLY permitted way to persist data. Direct `document.update()` calls outside StoreManager are architectural violations.

**Two rules violations to fix:**

**A. Knowledge/language cost uses Intelligence as the linked attribute threshold (same formula as active skills)**

SR3e rule (from sr3e-skill-rules.md): Knowledge and language skills always use Intelligence as their linked attribute. Cost is `rating ≤ Intelligence → 1 SP`, `rating > Intelligence → 2 SP` — the same formula as active skills.

The original editors had the correct formula but read `linkedAttribute` from the skill item data, which can be unreliable (SkillModel defaults to empty). Fix: hardcode Intelligence as the linked attribute for knowledge/language editors.

In `KnowledgeSkillEditorApp.svelte`:
- Replace generic `linkedAttribute = skill.system.knowledgeSkill.linkedAttribute` with hardcoded `attrs["intelligence"]`
- Keep cost formula: `cost = $valueStore < linkedAttrRating ? 1 : 2`
- Keep refund formula: `refund = $valueStore > linkedAttrRating ? 2 : 1`
- Keep delete refund loop: `for (let i = 1; i <= baseRating; i++) refund += i <= linkedAttrRating ? 1 : 2`

In `LanguageSkillEditorApp.svelte`: same fixes.

**B. Recalculate knowledge/language pools from Intelligence when attributes are locked**

SR3e rule: `knowledgePoints = Intelligence × 5`, `languagePoints = floor(Intelligence × 1.5)`.
These must be calculated from the FINAL intelligence value AFTER the player has spent all attribute points.
The correct trigger is when `attributeAssignmentLocked` transitions to `true`.

In `AttributePointsState.svelte`:
- Add `intelligenceStore = storeManager.GetRWStore<number>(actor, "attributes.intelligence.value")`
- `knowledgePoints` and `languagePoints` stores are already present (added in Bug 5 fix)
- In the `$effect` that handles attribute lock confirmation, BEFORE calling `attributeLocked.update(() => true)`, write the calculated pool values:
  ```ts
  const finalInt = $intelligenceStore ?? 1;
  $knowledgePoints = finalInt * 5;
  $languagePoints = Math.floor(finalInt * 1.5);
  await attributeLocked.update(() => true);
  ```
- These writes go through StoreManager (IStoreManager rule compliant)

In `CharacterInitializer.ts`:
- Remove the lines setting `"system.creation.knowledgePoints"` and `"system.creation.languagePoints"` from the `updateData` object — they'll now be set at attribute-lock time, not at character init time
- They will be 0 (schema default) until the player locks attributes

**Svelte 5 runes syntax reminder:**
- All stores accessed via `$storeName` subscription syntax
- Writes via `$storeName = value` (triggers document.update through StoreManager)
- `$effect(() => { ... })` for side effects
- `$derived(expression)` for computed values
- `$props()` for component props
