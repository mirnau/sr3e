  <action>
`SkillSpendingService` currently has the correct skeleton but uses:
- Flat 1 CP cost for all skills (wrong for active skills)
- `pointsService.getRemainingSkillPoints(actor, category)` which reads stale `actor.system` (stale read anti-pattern)
- No handling of `linkedAttrRating` for active skill cost

**Rewrite SkillSpendingService with the following API:**

```ts
type SkillCategory = "active" | "knowledge" | "language";

// Public methods the editors will call:

canIncrease(actor: Actor, skill: Item, category: SkillCategory, linkedAttrRating: number): boolean
// All skill types: remainingPoints >= cost where cost = currentValue < linkedAttrRating ? 1 : 2

increase(actor: Actor, skill: Item, category: SkillCategory, linkedAttrRating: number): void
// All skill types: valueStore.set(currentValue + 1); poolStore.set(pool - cost)
// cost = currentValue < linkedAttrRating ? 1 : 2

canDecrease(actor: Actor, skill: Item, category: SkillCategory): boolean
// currentValue > 0

decrease(actor: Actor, skill: Item, category: SkillCategory, linkedAttrRating: number): void
// All skill types: refund = currentValue > linkedAttrRating ? 2 : 1; valueStore.set(currentValue - 1); poolStore.set(pool + refund)

deleteWithRefund(actor: Actor, skill: Item, category: SkillCategory, linkedAttrRating: number): Promise<void>
// 1. Snapshot refund BEFORE delete: sum costs per level using linked-attr formula for all skill types
// 2. await actor.deleteEmbeddedDocuments("Item", [skill.id!])
// 3. After delete: poolStore.set(snapshotValue) (delete-first-then-refund pattern)
```

**Why explicit `category` param (NOT reading `skill.system.skillType`):**
Plan 01 fixes editor routing by passing `category` from SkillCard through SkillEditorApp — it does NOT fix the `skillType` field on the item data (which defaults to `"active"` for all skill items via `SkillModel.ts`). Reading `skillType` here would reintroduce the same routing bug at the service layer. Category must always be passed explicitly from the caller.

**Implementation rules:**
- Use `StoreManager.Instance.GetRWStore<number>(document, path)` to get stores
- Use `get(store)` from `svelte/store` to read current values (never `actor.system`)
- Use `store.set(value)` to write (goes through IStoreManager → triggers `document.update()`)
- Pool path map: `active → creation.activePoints`, `knowledge → creation.knowledgePoints`, `language → creation.languagePoints`
- Skill value path map: `active → activeSkill.value`, `knowledge → knowledgeSkill.value`, `language → languageSkill.value`
- Specialization-aware refund in `deleteWithRefund`:
  - `specializationsStore = StoreManager.Instance.GetRWStore(skill, \`${skillKey}.specializations\`)`
  - `baseRating = get(specializationsStore).length > 0 ? currentValue + 1 : currentValue`
  - All skill types: sum costs per level using linked-attr formula (`i <= linkedAttrRating ? 1 : 2`)

**Do NOT use `pointsService.getRemainingSkillPoints(actor, category)`** — it reads stale `actor.system`. Read directly from the store via `get(poolStore)`.

**Important:** The service does NOT call `storeManager.Subscribe(actor)` or `storeManager.Subscribe(skill)` — that is the component's responsibility. The service just gets stores and reads/writes them.

**Pattern for each editor:**

Each editor file implicitly knows its own category. Declare it as a constant and pass it to every service call — do NOT read it from `skill.system.skillType` (that field defaults to `"active"` on all items and is not reliable).

```ts
import { SkillSpendingService } from "../../../services/character-creation/SkillSpendingService";
import type { SkillCategory } from "../../../services/character-creation/SkillSpendingService";

const spendingService = SkillSpendingService.Instance();

// Hardcoded per editor file:
// ActiveSkillEditorApp:   const SKILL_CATEGORY: SkillCategory = "active";
// KnowledgeSkillEditorApp: const SKILL_CATEGORY: SkillCategory = "knowledge";
// LanguageSkillEditorApp:  const SKILL_CATEGORY: SkillCategory = "language";

// linkedAttrRating: keep the existing computation at mount time in ActiveSkillEditorApp (safe — attrs locked during skill phase)
// For knowledge/language editors: linkedAttrRating removed entirely (flat cost, no linked-attr needed)
// Pass 0 for linkedAttrRating in knowledge/language calls (service ignores it for non-active categories)

function increment(): void {
    if (!$isCreation) return; // Phase 3: karma shopping
    if (!spendingService.canIncrease(actor, skill, SKILL_CATEGORY, linkedAttrRating)) return;
    spendingService.increase(actor, skill, SKILL_CATEGORY, linkedAttrRating);
}

function decrement(): void {
    if (!$isCreation) return; // Phase 3: karma shopping
    if (!spendingService.canDecrease(actor, skill, SKILL_CATEGORY)) return;
    spendingService.decrease(actor, skill, SKILL_CATEGORY, linkedAttrRating);
}

async function deleteSkill(): Promise<void> {
    const confirmed = await foundry.applications.api.DialogV2.confirm({ ... });
    if (confirmed) {
        if ($isCreation) {
            await spendingService.deleteWithRefund(actor, skill, SKILL_CATEGORY, linkedAttrRating);
        } else {
            await actor.deleteEmbeddedDocuments("Item", [skill.id!]);
        }
        app?.close();
    }
}
```

**What to KEEP in each editor:**
- `storeManager.Subscribe(actor)` and `storeManager.Subscribe(skill)` with `onDestroy` cleanup
- `isCreation` flag store
- `valueStore` (for display)
- `specializationsStore` (for display and specialization UI interactions)
- `readwriteStore` in LanguageSkillEditorApp (for display + readWrite derived)
- `addSpecialization()` and `deleteSpecialization()` (specialization UI — no point cost at creation)
- All template/markup unchanged

**What to REMOVE from each editor:**
- Inline `increment()` and `decrement()` logic (replace with service delegation)
- Inline `deleteSkill()` refund calculation (replace with service delegation)
- `activePoints`/`knowledgePoints`/`languagePoints` stores — the service manages those
- `availableXPoints` derived (no longer needed in the component)
- `linkedAttrRating`/`linkedAttribute` variables in KnowledgeSkillEditorApp and LanguageSkillEditorApp (no longer needed — flat cost, pass 0 for linkedAttrRating)
- Keep `linkedAttrRating` in ActiveSkillEditorApp ONLY (still needed for active skill cost calculation)
- Also export `SkillCategory` type from `SkillSpendingService.ts` so editors can import it

**For specializations:**
Keep existing `addSpecialization()` and `deleteSpecialization()` inline (these are free at creation — just manipulate the array and adjust the base value by ±1). The service handles delete-with-refund which already accounts for specialization existence.

**Svelte 5 runes syntax:** All reactive values use `$derived()`, effects use `$effect()`, props use `$props()`. No legacy patterns.
