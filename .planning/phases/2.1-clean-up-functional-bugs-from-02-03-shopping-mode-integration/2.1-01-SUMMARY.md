### Bug 1 ŌĆö Attribute points not consumed or refunded on buy/return

**Root cause:** `AttributePointsState.svelte` called `storeManager.GetROStore(actor)` with no
`dataPath` argument. This created a store at `"system.undefined"` that never received updates,
so the `pointList` `$derived` never re-ran. The display was frozen at the initial value.

**Fix:** Replaced broken `GetROStore(actor)` with `GetRWStore<number>(actor, "creation.attributePoints")`.
The `pointList` derived now uses `$creationPointsStore` directly as both the reactive trigger
and the displayed value. The auto-lock `$effect` was updated to use the same store.

**Files:** `module/ui/actors/injections/AttributePointsState.svelte`

---

### Bug 2 ŌĆö Racial attribute maximum not enforced (could buy past 6 for Human)

**Root cause:** `AttributeSpendingService` read from `actor.system` to validate can-increase and
read current values in increase/decrease. `actor.system` is stale until `document.update()`
resolves asynchronously ŌĆö but Svelte re-renders synchronously after `store.set()`. Validation
saw a stale value and allowed spending past the racial cap.

**Fix:** All four methods (`canIncreaseAttribute`, `canDecreaseAttribute`, `increaseAttribute`,
`decreaseAttribute`) now use `get(store)` from `svelte/store` to read the current reactive value.
Removed the now-unused `CreationPointsService` import and `#getAttributeValue()` helper.

**Files:** `module/services/character-creation/AttributeSpendingService.ts`

### Bug 3 ŌĆö Knowledge/language skill pools were priority-based, not intelligence-derived

> ŌÜĀ’ĖÅ **This bug fix was itself incorrect and was superseded ŌĆö see Bug 4 below.**

**Root cause:** Knowledge and language pools were computed as `skillPoints ├Ś 0.25` and
`skillPoints ├Ś 0.15` (static priority distribution) and stored as remaining values.
SR3e rule: knowledge = `Int ├Ś 5`, language = `floor(Int ├Ś 1.5)` ŌĆö both dynamic as
intelligence is spent during attribute assignment.

**Attempted fix (SUPERSEDED):** Switched to "computed total minus stored spent" model using
`knowledgeSpent`/`languageSpent` fields. This was wrong ŌĆö those fields do not exist in the
`Creation` data model schema (`Creation.ts`). Foundry silently discards writes to undefined
schema fields, so spending and reimbursement never persisted.

**Files affected:**
- `module/services/character-creation/CharacterInitializer.ts`
- `module/services/character-creation/CreationPointsService.ts`
- `module/services/character-creation/SkillSpendingService.ts`
- `module/ui/actors/injections/AttributePointsState.svelte`

### Bug 4 ŌĆö Knowledge/language spending/reimbursement never persisted (supersedes Bug 3)

**Root cause:** `Creation.ts` defines `knowledgePoints` and `languagePoints` as remaining-pool
fields (same pattern as `activePoints`). The Bug 3 fix introduced non-existent fields
`knowledgeSpent`/`languageSpent`. Foundry's `TypeDataModel` silently discards writes to fields
not in the schema, so all knowledge/language point changes were lost on the next document sync.

**The correct model ŌĆö all three skill categories use the same remaining-pool pattern:**

| Field | Type | Meaning |
|---|---|---|
| `creation.activePoints` | remaining | Points left to spend on active skills |
| `creation.knowledgePoints` | remaining | Points left to spend on knowledge skills |
| `creation.languagePoints` | remaining | Points left to spend on language skills |

**Do not introduce `knowledgeSpent`, `languageSpent`, or `activeSpent` ŌĆö these fields do not
exist in `Creation.ts` and Foundry will silently discard any writes to them.**

**Fix:**
- `CharacterInitializer.ts`: initialise `knowledgePoints` and `languagePoints` from the priority
  table via `getDistributedSkillPoints()`, same as `activePoints`.
- `CreationPointsService.ts`: `getRemainingSkillPoints("knowledge/language")` returns
  `system.creation.knowledgePoints/languagePoints` directly. Removed intel-derived calculation
  and now-unused `StoreManager`/`get` imports.
- `SkillSpendingService.ts`: unified all three categories ŌĆö spending decrements the pool field,
  refunding increments it. Removed the active/knowledge-language branching.
- `KnowledgeSkillEditorApp.svelte`: `knowledgeSpent` ŌåÆ `knowledgePoints`; direction flipped
  (decrement on buy, increment on refund); removed unused `intelligenceStore`.
- `LanguageSkillEditorApp.svelte`: same as above for `languagePoints`.
- `SkillPointsState.svelte`: store names updated to `knowledgePoints`/`languagePoints`.

**Files:**
- `module/models/actors/actor-components/Creation.ts` ŌĆö source of truth, do not add spent fields
- `module/services/character-creation/CharacterInitializer.ts`
- `module/services/character-creation/CreationPointsService.ts`
- `module/services/character-creation/SkillSpendingService.ts`
- `module/ui/items/skill-editor/KnowledgeSkillEditorApp.svelte`
- `module/ui/items/skill-editor/LanguageSkillEditorApp.svelte`
- `module/ui/actors/injections/SkillPointsState.svelte`

### Bug 5 ŌĆö Skill point display in both panels showed stale values (reactivity broken)

**Root cause:** Both `SkillPointsState.svelte` and `AttributePointsState.svelte` used
`pointsService.getRemainingSkillPoints(actor, category)` inside `$derived`. The `$derived`
re-ran on store changes (correct triggers) but the values it returned came from
`actor.system` reads inside the service ŌĆö which is async-stale until `actor.update()` resolves.
So the display always showed the pre-change value.

`AttributePointsState.svelte` also subscribed to `intelligenceStore` as an extra trigger for
knowledge/language pools (a leftover from when those pools were intel-derived). This was no longer
needed once knowledge/language became fixed priority-based pools (Bug 4 fix).

**Fix:** Removed `CreationPointsService` dependency from both components entirely. Both now get
`activePoints`, `knowledgePoints`, and `languagePoints` stores directly from `StoreManager` and
read `$store ?? 0` inside `$derived`. No service intermediary ŌĆö no stale reads.
`intelligenceStore` removed from `AttributePointsState.svelte`.

**Pattern to follow for all reactive point displays:**
```ts
// Ō£ģ Correct ŌĆö reads live reactive value
const activePoints = storeManager.GetRWStore<number>(actor, "creation.activePoints");
value: $activePoints ?? 0

// ŌØī Wrong ŌĆö reads actor.system which is stale
value: pointsService.getRemainingSkillPoints(actor, "active")
```

**Files:**
- `module/ui/actors/injections/SkillPointsState.svelte`
- `module/ui/actors/injections/AttributePointsState.svelte`

### Visual Bug 1 ŌĆö Movement, DicePools, Health stat cards larger than attribute cards

**Root cause:** `.stat-card-wrapper` had only `min-width: 100px` (no height). In a flex container
with `align-items: stretch`, cards could grow vertically. Long labels like "COMBAT POOL" could
also push cards wider than 100px.

**Fix:**
- `statcard.scss`: Added explicit `width: var(--attribute-card-width)` (100px) and
  `height: var(--attribute-card-height)` (4rem) to both `.stat-card-wrapper` and
  `.attribute-grid-item`. Added `white-space: nowrap; overflow: hidden; text-overflow: ellipsis`
  to `.attribute-label`.
- `Health.svelte`: Replaced `<div class="health-card-container"><div class="stat-grid single-column">`
  (CSS Grid with `grid-template-columns: 100%` ŌĆö could stretch cards to container width) with
  `<div class="stat-card-grid">` (same container class as Movement/DicePools).

**Files:** `styles/statcard.scss`, `module/ui/actors/actor-components/Health.svelte`

### Visual Bug 2 ŌĆö Stat cards (Combat, Walking, Running) stretched wider than attribute cards

**Root cause:** Attribute cards get their width from the Packery column algorithm:
`floor((containerWidth ŌłÆ gutters) / columnCount)` Ōēł 111px in a typical 228px sheet column.
Stat cards had `width: 100px` fixed, so they were narrower and looked proportionally different.
Additionally, a single card in a flex row (like COMBAT alone in DicePools) had no siblings to
fill the row, making it look isolated and oddly sized compared to the dense attribute grid.

**Fix:** Replaced the flexbox `.stat-card-grid` with CSS Grid using
`grid-template-columns: repeat(auto-fill, minmax(var(--attribute-card-width), 1fr))`.
This replicates the Packery math exactly ŌĆö in a 228px container with 6px gap, two 100px
min-width columns auto-fill to `(228 ŌłÆ 6) / 2 = 111px` each ŌĆö matching attribute card width.
A single card (COMBAT) still gets one column = 111px, not stretched to full width.
`.stat-card-wrapper` changed to `width: 100%` to fill the grid cell.

**Files:** `styles/statcard.scss`

## Decisions Made

- **Reactive store reads over actor.system** ŌĆö all creation services that need "current" values
  during a synchronous render must use `get(store)`, since `actor.system` is async-stale.
  UI components must read display values directly from stores, never via service methods that
  read `actor.system` ŌĆö even if the `$derived` trigger is a store subscription.
- **Remaining pool for all skill categories** ŌĆö `activePoints`, `knowledgePoints`, and
  `languagePoints` all follow the same pattern: initialised from the priority table, decremented
  on spend, incremented on refund. Do not use spent-counter fields (`*Spent`) ŌĆö they are not in
  the `Creation.ts` schema and Foundry will silently discard writes to them.
- **CSS Grid auto-fill matches Packery math** ŌĆö `repeat(auto-fill, minmax(100px, 1fr))` is
  mathematically equivalent to Packery's column calculation and requires no JS.
