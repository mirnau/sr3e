  <name>Task 1: Debloat Skills.svelte — remove itemsUpdateTick and Foundry hooks</name>
  <files>module/ui/actors/actor-components/Skills.svelte</files>
  <action>
Skills.svelte has the `itemsUpdateTick` pattern with manual Foundry hook registrations — the same pattern that was removed from DicePools.svelte. Apply the same fix.

**Anti-patterns to remove:**

1. **`itemsUpdateTick` $state** (line 11): Remove `let itemsUpdateTick = $state(0)`.

2. **Manual Foundry hook registrations in $effect** (lines 14-46): The entire `$effect` block registers `collection.on("update/create/delete", bump)` and `Hooks.on("updateItem/createItem/deleteItem", bumpHook)`. Remove this entire `$effect`. StoreManager already subscribes to `actorSystemRecalculated` which fires when items change — no manual hook registration needed.

3. **Move Subscribe/Unsubscribe**: Replace the removed `$effect` with top-level `storeManager.Subscribe(actor)`. Add `import { onDestroy } from "svelte"` and `onDestroy(() => storeManager.Unsubscribe(actor))`.

4. **`itemsUpdateTick >= 0` in $derived expressions** (lines 47-80): The `activeSkills`, `knowledgeSkills`, `languageSkills` derived values use `itemsUpdateTick >= 0 ? [...] : []` to force reactivity. Remove this trick — `$derived` reading `actor.items` will re-run when actor data changes via StoreManager. Simplify:
   ```ts
   const activeSkills = $derived(
      [...(actor.items ?? [])]
         .filter((item: any) => item.type === "skill" && item.system?.skillType === "active")
         .sort((a: any, b: any) => a.name.localeCompare(b.name))
   );
   ```
   Same simplification for `knowledgeSkills` and `languageSkills`.

5. **Required actor prop** (line 8): Change `let { actor = null } = $props()` to `let { actor }: { actor: Actor } = $props()`.

6. **`activeTab` $state** (line 12): `activeTab` controls tab selection — this is legitimate local UI state. Keep `let activeTab = $state<"active" | "knowledge" | "language">("active")`. This is NOT a store, it's component-local state.
  </action>
  <verify>File compiles. No `itemsUpdateTick`, no `Hooks.on`, no `collection.on`. Actor prop is required. `activeSkills`/`knowledgeSkills`/`languageSkills` are clean `$derived` expressions.</verify>
  <done>
  - `itemsUpdateTick` removed
  - All Foundry hook registrations removed
  - `storeManager.Subscribe` at top level, `onDestroy` for cleanup
  - Three skill list deriveds are clean (no tick trick)
  - Required actor prop
  </done>

<task type="auto">
  <name>Task 2: Debloat SkillCard.svelte — stores out of $effect</name>
  <files>module/ui/actors/actor-components/skills/SkillCard.svelte</files>
  <action>
SkillCard.svelte has the classic $state+$effect store initialization pattern. It subscribes to BOTH `item` (for skill data) and `actor` (for flag stores).

**Anti-patterns to fix:**

1. **5 stores in $state+$effect** (lines 21-46): `valueStore`, `specializationsStore`, `readWriteStore`, `isShoppingState`, `isCharacterCreation` are `$state` variables initialized inside `$effect`. Move to top-level `const`:
   ```ts
   storeManager.Subscribe(item);
   storeManager.Subscribe(actor);

   const valueStore = storeManager.GetRWStore<number>(item, `${skillKey}.value`);
   const specializationsStore = storeManager.GetRWStore<Specialization[]>(item, `${skillKey}.specializations`);
   const readWriteStore = category === "language"
      ? storeManager.GetRWStore<number>(item, "languageSkill.readwrite.value")
      : null;
   const isShoppingState = storeManager.GetFlagStore<boolean>(actor, "isShoppingState", false);
   const isCharacterCreation = storeManager.GetFlagStore<boolean>(actor, "isCharacterCreation", false);
   ```
   Note: `category` is a prop that never changes after mount, so the conditional at top-level is safe.

2. **Unsubscribe in $effect cleanup** (lines 42-45): Replace with `onDestroy(() => { storeManager.Unsubscribe(item); storeManager.Unsubscribe(actor); })`. Add `import { onDestroy } from "svelte"`.

3. **Required actor prop** (lines 7-13): Change `actor: Actor | null` → `actor: Actor`. Remove `= null` default. Actor is always passed from SkillsActive/Knowledge/Language which receive it as a required prop (after plan 01 fixes).

4. **`valueStore ? $valueStore : 0` in template** (lines 85, 111): Simplify to `{$valueStore}` — store is always initialized.

5. **`$specializationsStore?.length ?? 0`** (lines 87, 130): Since `specializationsStore` is `Writable<Specialization[]>`, `$specializationsStore` is always an array. Simplify condition to `($specializationsStore?.length ?? 0) > 0` → `$specializationsStore.length > 0`. Remove `?? 0`.

6. **`($specializationsStore ?? [])` in `{#each}` loops**: Simplify to `{#each $specializationsStore as spec}` — store provides default array.

7. **`$readWriteStore ?? 0` in template** (line 125): Since readWriteStore may be `null` (for non-language categories), keep the null check: `{#if category === "language" && readWriteStore && $readWriteStore > 0}`. But replace `$readWriteStore ?? 0` with `$readWriteStore`.

8. **Remove `Writable` type import** if no longer used after converting from `$state<Writable<...> | null>`.
  </action>
  <verify>File compiles. No `$state` store variables. Both Subscribe calls at top level. `onDestroy` handles both Unsubscribes. Actor prop is `Actor` (not nullable).</verify>

  - Required actor prop
